# 画面設計書

対象: `frontend/`(React + Vite + react-router-dom)。実装済みの画面・挙動を記載する(未実装機能は「未実装」と明記)。

## 1. 画面一覧

| # | 画面名 | URL(パス) | コンポーネント(feature) | 認証 |
|---|---|---|---|---|
| 1 | 商品選択画面 | `/` | `features/product-select` | 不要 |
| 2 | 注文入力画面 | `/order/:productId` | `features/order-form` | 不要 |
| 3 | 注文完了画面 | `/confirm` | `features/confirm` | 不要 |
| 4 | 管理者ログイン画面 | `/admin/*` 配下のいずれか未ログイン時共通 | `features/admin-auth`(`LoginForm`) | - |
| 5 | 管理者注文一覧画面 | `/admin` | `features/admin-orders` | ユーザーID・パスワード |
| 6 | 商品カテゴリ管理画面 | `/admin/master/product-categories` | `features/product-categories` | ユーザーID・パスワード |
| 7 | 商品マスタ管理画面 | `/admin/master/products` | `features/product-master` | ユーザーID・パスワード |

ルーティング定義: `frontend/src/route/index.tsx`(`main.tsx`は`App.tsx`を呼ぶだけ、`App.tsx`が`BrowserRouter`+`AppRoutes`を配置)。`/admin`配下の3画面(5〜7)はReact Routerのネストルートで`components/layout/AdminGate.tsx`配下に置かれており、`AdminGate`がログイン状態を判定して画面4(ログインフォーム)を返すか、`AdminLayout`(サイドバー付きレイアウト)+`<Outlet/>`で実際のページを返すかを切り替える。ログイン状態は`features/admin-auth`の`AdminAuthProvider`(React Context)で3画面共通に保持されるため、`/admin`でログインした後に他の管理画面へ移動しても再ログインは不要。

## 2. 画面遷移図

```
[1. 商品選択画面] --商品カードをクリック--> [2. 注文入力画面]
[2. 注文入力画面] --「この内容で注文する」送信成功--> [3. 注文完了画面]
[3. 注文完了画面] --「商品一覧に戻る」--> [1. 商品選択画面]

[4. 管理者ログイン画面] --ログイン成功--> [5. 管理者注文一覧画面](既定の遷移先)
[5/6/7 いずれかの管理画面] --サイドバーのメニュークリック--> [5/6/7 の別の画面](再ログイン不要)
[5/6/7 いずれかの管理画面] --サイドバー下部「ログアウト」--> [4. 管理者ログイン画面]
```

- 画面2→3の遷移では、`react-router`の`navigate('/confirm', { state: { orderId } })`でorderIdを渡している。`/confirm`に直接アクセスした場合(リロード等)は`orderId`が失われ、注文番号非表示のまま完了メッセージのみ表示される。
- 商品選択画面(1)から注文入力画面(2)へは、`productId`をURLパスパラメータとして渡す。

## 3. 共通レイアウト

- 全画面共通で `.page` クラス(最大幅640px、中央寄せ)を使用。管理画面(5〜7)は `.page-wide`(最大幅920px)を追加適用し、検索バー・一覧・テーブルを横広に表示する。
- ヘッダー部は `kicker`(小さいラベル文言)+`h1` の組み合わせで統一。
- フォント: 見出し `Shippori Mincho`、本文 `Zen Kaku Gothic New`。
- エラーメッセージは共通で `.error-box` に赤系背景で表示。
- 管理画面(5〜7)は共通で `components/layout/AdminLayout.tsx` による開閉可能なサイドバー付きレイアウトを持つ(詳細は7.1)。shadcn/uiの`Sidebar`コンポーネントを使用し、テーマはブランドカラー(テラコッタ系)にカスタマイズ済み。

## 4. 商品選択画面(ProductSelect)

- URL: `/`
- 目的: 注文可能な商品一覧を表示し、注文する商品を選ばせる。

### 表示内容

| 項目 | 内容 | データソース |
|---|---|---|
| 見出し | 「ご注文商品の選択」 | 固定文言 |
| 商品カード一覧 | 商品名・説明・参考価格(`¥{priceFrom}〜`) | `GET /products` |
| ローディング表示 | 「読み込み中...」 | API未応答中 |
| エラー表示 | APIエラー時のメッセージ | `.error-box` |

### 挙動

- 画面表示時(`useEffect`)に `GET /products` を呼び出し、商品一覧を取得する。
- 各商品カードはボタン要素。クリックすると `/order/:productId` へ遷移する。
- 商品データはDB(`products`テーブル)から取得され、モックではない。商品の追加・編集を行うUIは未実装(現状はDBへ直接投入するかマイグレーション追加が必要)。

## 5. 注文入力画面(OrderForm)

- URL: `/order/:productId`
- 目的: 選択した商品に対して、注文者情報・数量・デザインファイルを入力し注文を送信する。

### 入力項目

| 項目 | 入力種別 | 必須 | バリデーション(フロント) | 対応するAPIフィールド |
|---|---|---|---|---|
| お名前 | テキスト | ○ | HTML `required` | `customerName` |
| メールアドレス | メール | ○ | HTML `required` + `type=email` | `customerEmail` |
| 数量 | 数値 | ○ | `min=1` | `quantity` |
| ご要望・備考 | テキストエリア | 任意 | なし | `notes` |
| デザインデータ・イメージ画像 | ファイル(複数) | 任意 | 拡張子は`image/*, application/pdf`のみ選択可(input accept属性、強制力なし)。最大5点までフロント側でスライスして保持 | `files`(multipart) |

- 画面上部に選択中の商品名を表示(`GET /products`の結果から`productId`一致するものを検索して表示。商品個別取得APIは無く、一覧取得結果をフロントでフィルタしている)。
- 送信ボタン押下で `POST /orders`(multipart/form-data)を呼び出す。
- 送信中はボタンが「送信中...」表示になり、二重送信を防止(`disabled`)。
- 送信失敗時は `.error-box` にエラーメッセージを表示し、画面遷移しない。
- 送信成功時は `/confirm` へ遷移し、`orderId`を画面stateとして渡す。

### ファイル添付の制約(実装値)

- 選択できるファイルは最大5点(超過分はフロントで自動的に切り捨て)。
- サーバー側は1ファイルあたり最大20MBまで(`FilesInterceptor('files', 5, { limits: { fileSize: 20 * 1024 * 1024 } })`)。
- サーバー側でのファイル種別(拡張子・MIME)チェックは行っていない。フロントの`accept`属性はブラウザのファイル選択ダイアログの絞り込みに過ぎず、セキュリティ上の制御ではない。

## 6. 注文完了画面(Confirm)

- URL: `/confirm`
- 目的: 注文送信が完了したことをユーザーに伝える。

### 表示内容

- 「済」スタンプ風の丸バッジ
- 「ご注文ありがとうございました」の見出し
- 案内文言(内容確認後に支払い案内メールを送る旨)
- 注文番号(`location.state.orderId`が存在する場合のみ表示。直接URLアクセスやリロード時は非表示)
- 「商品一覧に戻る」リンク(`/`へ)

このページ自体はAPIを呼び出さない。表示専用の画面。

## 7. 管理者ログイン画面(AdminOrders / 未ログイン状態)

- URL: `/admin`(初期表示)
- 目的: 管理者がユーザーID・パスワードを入力し、注文管理機能にアクセスする。

### 入力項目

| 項目 | 入力種別 | 必須 |
|---|---|---|
| ユーザーID | テキスト(`autoComplete=username`) | ○ |
| パスワード | パスワード(`autoComplete=current-password`) | ○ |

### 挙動

- 「ログイン」ボタンは両フィールドが空の間は非活性。
- 送信すると、入力値をヘッダー(`x-admin-id` / `x-admin-password`)に載せて `GET /orders` を実行し、成功可否でログイン成否を判定する(専用のログインAPIは無く、注文一覧取得の成否を認証確認に流用している)。
- 失敗時(401等)は `.error-box` にエラーメッセージを表示し、ログイン画面のまま留まる。
- ログイン状態はReactのコンポーネントstateのみで保持しており、**画面リロードやタブを閉じると必ずログアウトされる**(トークン等の永続化は無し)。

## 8. 管理者注文一覧画面(AdminOrders / ログイン後状態)

- URL: `/admin`(ログイン成功後)
- 目的: 全注文の一覧確認、検索、ステータス変更、支払いリンク送信を行う。

### 8.1 検索条件(検索バー)

| 項目 | 入力種別 | 説明 |
|---|---|---|
| キーワード | テキスト | 顧客名・メールアドレス・商品名・注文IDのいずれかに部分一致(大文字小文字区別なし、`ILIKE`) |
| ステータス | セレクトボックス | 「すべて」または4ステータスのいずれか |
| 期間(開始) | 日付(date input) | `createdAt >= 指定日 00:00:00.000Z` |
| 期間(終了) | 日付(date input) | `createdAt <= 指定日 23:59:59.999Z` |

- 「検索」ボタンで `GET /orders` にクエリパラメータを付けて再取得。
- 「条件をリセット」で検索条件を全てクリアして再取得。
- 検索条件はAND結合。日付はJST変換等は行わず、UTC基準で日付境界を作っている点に注意(サーバーのタイムゾーン設定に依存)。

### 8.2 注文一覧表示

一覧はカード(行)形式で、1注文につき以下を表示する。

| 表示項目 | 内容 |
|---|---|
| 商品名 / 顧客名 | `{productName} / {customerName}様` |
| ステータスバッジ(兼変更用セレクト) | 4色に色分け(下表参照)。クリックしてその場でステータス変更可能 |
| 顧客メールアドレス・数量・注文日時 | `toLocaleString('ja-JP')`で表示 |
| 注文ID | UUID全体を表示 |
| 備考 | `notes`が存在する場合のみ表示 |
| 添付ファイル名 | `fileNames`をカンマ区切りで表示(ファイル名のみのテキスト表示。**ダウンロード・プレビュー機能は未実装**) |
| 支払いリンク入力欄 or 送信済みリンク表示 | ステータスが`payment_link_sent`以外なら入力フォーム、それ以外は送信済みリンクをテキスト表示 |

一覧の先頭に「◯件の注文」という件数表示がある。該当0件の場合は「条件に一致する注文がありません。」を表示。

### 8.3 ステータスバッジの色分け

| ステータス値 | 表示ラベル | バッジ色クラス |
|---|---|---|
| `new` | 新規注文 | `badge-new`(オレンジ系) |
| `reviewing` | 内容確認中 | `badge-reviewing`(黄色系) |
| `payment_link_sent` | メール送信済み | `badge-sent`(緑系) |
| `cancelled` | キャンセル | `badge-cancelled`(グレー系) |

このバッジ自体がセレクトボックスになっており、選択肢を変更すると即座に `PATCH /orders/:id/status` が呼ばれてステータスが更新される(確認ダイアログ等は無し)。ステータス遷移に業務ルール上の制約は無く、どのステータスからどのステータスへも自由に変更可能。

### 8.4 支払いリンク送信

- ステータスが`payment_link_sent`でない注文には、URL入力欄と「送信」ボタンを表示。
- 「送信」を押すと `POST /orders/:id/send-payment-link` を呼び出し、成功すると当該注文のステータスが強制的に`payment_link_sent`に変わり、顧客にメールが送信される(SMTP未設定の場合はサーバーログ出力のみでメールは送られない)。
- 送信済みの注文は入力欄の代わりに「送信済みリンク: {URL}」というテキストを表示する(リンクとしてクリック可能にはなっていない)。

### 8.5 ログアウト

- ヘッダー右側の「ログアウト」ボタンで、保持していた認証情報・注文一覧・パスワード入力値をクリアし、ログイン画面に戻す。サーバー側のセッション破棄処理は無い(そもそもサーバー側にセッション概念が無い)。

---

## 9. 未実装・既知の制約(画面まわり)

- 商品の追加・編集・削除画面は無い(閲覧のみ)。
- 添付ファイルのダウンロード・プレビューは管理画面から行えない(ファイル名の表示のみ)。
- 管理者ログイン状態は永続化されない(リロードで毎回ログインし直しが必要)。
- 注文の削除・キャンセル理由入力などの機能は無い(ステータスを`cancelled`にするだけ)。
- ページネーションは無く、`GET /orders`の結果を全件一覧表示する。
