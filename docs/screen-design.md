## 2. 画面遷移図

```
[1. カテゴリ選択画面] --カテゴリカードをクリック--> [2. カテゴリ別商品一覧画面]
[2. カテゴリ別商品一覧画面] --商品カードをクリック--> [3. 商品詳細画面]
[3. 商品詳細画面] --数量を選び「注文情報の入力へ」--> [4. 注文入力画面]
[4. 注文入力画面] --「この内容で注文する」送信成功--> [5. 注文完了画面]
[5. 注文完了画面] --「商品一覧に戻る」--> [1. カテゴリ選択画面]

[6. 管理者ログイン画面] --ログイン成功--> [7. 管理者注文一覧画面](既定の遷移先)
[7. 管理者注文一覧画面] --注文カードを選択--> [8. 注文詳細画面]
[8. 注文詳細画面] --「注文一覧へ戻る」--> [7. 管理者注文一覧画面]
[7〜11 いずれかの管理画面] --サイドバー下部「ログアウト」--> [6. 管理者ログイン画面]
```

- 画面4→5の遷移では、`react-router`の`navigate('/confirm', { state: { orderId } })`でorderIdを渡している。`/confirm`に直接アクセスした場合(リロード等)は`orderId`が失われ、注文番号非表示のまま完了メッセージのみ表示される。
- 画面3→4では`productId`をパス、選択数量を`quantity`クエリパラメータで渡す。不正な数量または未指定の場合は1を初期値にする。

## 3. 共通レイアウト

- 全画面共通で `.page` クラス(最大幅640px、中央寄せ)を使用。画面幅560px以下では、最下部がブラウザUIや端末のセーフエリアに隠れないよう、全`.page`に140pxとセーフエリア分の下余白を確保する。管理者注文一覧・注文詳細画面は`.page-wide`を追加し、最大幅を固定せず、管理画面の利用可能領域から左右`clamp(16px, 2.5vw, 32px)`の余白を引いた幅で表示する。これにより内容量に関係なく両画面の横幅が揃い、ディスプレイ幅に追従する。商品カテゴリ・商品マスタは共通の`.page-master.master-management-page`を使い、左右`clamp(16px, 2.5vw, 32px)`の余白、ヘッダー、検索領域、テーブルviewport、編集ダイアログのサイズと余白を統一する。
- マスタ管理の画面枠は`components/master/MasterPageLayout`、横スクロール領域は`MasterTableViewport`、新規作成・編集ダイアログは`MasterFormDialogLayout`で共通化する。画面ごとに変わるのは検索条件、テーブルのカラムと最低幅、入力項目のみ。両テーブルは`table-layout: auto`とカラムごとの最低幅を使い、画面が狭い場合は共通viewport内だけを横スクロールさせる。
- ヘッダー部は `kicker`(小さいラベル文言)+`h1` の組み合わせで統一。
- フォント: 見出し `Shippori Mincho`、本文 `Zen Kaku Gothic New`。
- エラーメッセージは共通で `.error-box` に赤系背景で表示。
- 管理画面(7〜11)は共通で `components/layout/AdminLayout.tsx` による開閉可能なサイドバー付きレイアウトを持つ(詳細は9.1)。shadcn/uiの`Sidebar`コンポーネントを使用し、テーマはブランドカラー(テラコッタ系)にカスタマイズ済み。

## 4. カテゴリ選択画面(ProductSelect)

- URL: `/`
- 目的: 最初に注文したい紙製品のカテゴリを1つ選ばせる。

### 表示内容

| 項目 | 内容 | データソース |
|---|---|---|
| 見出し | 「カテゴリを選ぶ」 | 固定文言 |
| カテゴリカード一覧 | 画像・カテゴリ名・「商品を見る」リンク表現 | `GET /product-categories` |
| カード表示切替 | 「1列」「2列」のセグメントボタン。初期値は2列 | 画面内state |
| ローディング表示 | 「読み込み中...」 | API未応答中 |
| エラー表示 | APIエラー時のメッセージ | `.error-box` |

### 挙動

- 画面表示時に公開API `GET /product-categories` を呼び出す。
- 各カードはボタン要素。クリックすると `/categories/:categoryId/products` へ遷移する。
- 「1列」「2列」でカテゴリカードの列数を切り替える。選択中ボタンは色付き表示と`aria-pressed=true`になり、画面幅が狭い場合も選択した列数を維持する。
- カテゴリカードの画像はカテゴリ名の行数に関係なく幅100%・アスペクト比4:3を維持する。カテゴリ名には2行分の高さを確保し、1行または2行の場合も画像と「商品を見る」リンクの位置を揃える。3行以上の場合は画像を縮めずカード全体を縦に拡張する。
- 最終カードが画面下端や端末のセーフエリアに隠れないよう、画面下部に140pxとセーフエリア分の余白を確保する。
- 画像未登録時は紙のテクスチャ風プレースホルダーを表示する。

## 5. カテゴリ別商品一覧画面(CategoryProducts)

- URL: `/categories/:categoryId/products`
- `GET /products`と`GET /product-categories`を並行取得し、`productCategoryId`がURLのカテゴリIDと一致する商品のみをカード形式で表示する。
- 商品カードには商品画像・商品名・説明(2行まで)・参考価格を表示する。画像未登録時はプレースホルダーを表示する。
- 「1列」「2列」のセグメントボタンで商品カードの列数を切り替える。初期値は2列で、画面幅が狭い場合も選択した列数を維持する。
- 最終カードが画面下端や端末のセーフエリアに隠れないよう、画面下部に140pxとセーフエリア分の余白を確保する。
- 商品カードをクリックすると`/products/:productId`へ遷移する。該当商品が0件の場合は空状態メッセージを表示する。

## 6. 商品詳細画面(ProductDetail)

- URL: `/products/:productId`
- `GET /products/:productId`で商品を取得し、上から商品画像、カテゴリ名、商品名、参考価格、説明、数量入力、注文ボタンの順に表示する。
- 数量は1以上の整数。入力中は現在値を消して空欄にできるため、「1」を削除してから任意の数量を入力できる。空欄・0以下・小数の場合は「注文情報の入力へ」ボタンを非活性にし、有効な数量でのボタン押下時に`/order/:productId?quantity=:quantity`へ遷移する。
- 「カテゴリの商品一覧へ」で選択商品のカテゴリ別一覧へ戻る。

## 7. 注文入力画面(OrderForm)

- URL: `/order/:productId`
- 目的: 選択した商品に対して、注文者情報・数量・デザインファイルを入力し注文を送信する。

### 入力項目

| 項目 | 入力種別 | 必須 | バリデーション(フロント) | 対応するAPIフィールド |
|---|---|---|---|---|
| お名前 | テキスト | ○ | HTML `required` | `customerName` |
| メールアドレス | メール | ○ | HTML `required` + `type=email` | `customerEmail` |
| 電話番号 | 電話番号 | 任意 | `type=tel` | `customerPhone` |
| 数量 | 数値 | ○ | `min=1` | `quantity` |
| ご要望・備考 | テキストエリア | 任意 | なし | `notes` |
| デザインデータ・イメージ画像 | ファイル(複数) | 任意 | 拡張子は`image/*, application/pdf`のみ選択可(input accept属性、強制力なし)。最大5点までフロント側でスライスして保持。HEIC/HEIFはサーバーでJPEGへ変換 | `files`(multipart) |

- 画面上部に選択中の商品名を表示(`GET /products/:productId`で取得)。商品詳細で選択した数量を初期値にする。
- 送信ボタン押下で `POST /orders`(multipart/form-data)を呼び出す。
- 送信中はボタンが「送信中...」表示になり、二重送信を防止(`disabled`)。
- 送信失敗時は `.error-box` にエラーメッセージを表示し、画面遷移しない。
- 送信成功時は `/confirm` へ遷移し、`orderId`を画面stateとして渡す。

### ファイル添付の制約(実装値)

- 選択できるファイルは最大5点(超過分はフロントで自動的に切り捨て)。
- サーバー側は1ファイルあたり最大20MBまで(`FilesInterceptor('files', 5, { limits: { fileSize: 20 * 1024 * 1024 } })`)。
- サーバー側でのファイル種別(拡張子・MIME)チェックは行っていない。フロントの`accept`属性はブラウザのファイル選択ダイアログの絞り込みに過ぎず、セキュリティ上の制御ではない。
- HEIC/HEIF画像はブラウザ表示互換性のため、アップロード時にJPEGへ変換して保存する。変換失敗時は注文を作成せずエラーを表示する。

## 8. 注文完了画面(Confirm)

- URL: `/confirm`
- 目的: 注文送信が完了したことをユーザーに伝える。

### 表示内容

- 「済」スタンプ風の丸バッジ
- 「ご注文ありがとうございました」の見出し
- 案内文言(内容確認後に支払い案内メールを送る旨)
- 注文番号(`location.state.orderId`が存在する場合のみ表示。直接URLアクセスやリロード時は非表示)
- 「商品一覧に戻る」リンク(`/`へ)

注文作成APIの処理中に、管理者への新規注文通知と、注文者への注文ID付き受付完了メールを送信する。メール送信が失敗しても保存済みの注文は成立し、この完了画面へ遷移する。このページ自体はAPIを呼び出さない表示専用画面。

## 9. 管理者ログイン画面(features/admin-auth の LoginForm)

- URL: `/admin/*`配下のいずれかに未ログインでアクセスした場合共通(`components/layout/AdminGate.tsx`が判定して表示)
- 目的: 管理者がユーザーID・パスワードを入力し、管理画面(注文管理・マスタ管理)にアクセスする。

### 入力項目

| 項目 | 入力種別 | 必須 |
|---|---|---|
| ユーザーID | テキスト(`autoComplete=username`) | ○ |
| パスワード | パスワード(`autoComplete=current-password`) | ○ |

### 挙動

- 「ログイン」ボタンは両フィールドが空の間は非活性。
- 送信すると、入力値をヘッダー(`x-admin-id` / `x-admin-password`)に載せて `POST /admin/login` を呼び出し、成功可否でログイン成否を判定する(注文一覧取得への副作用的な流用ではなく、専用のログイン確認APIになった)。
- 成功すると`AdminAuthProvider`(React Context)に認証情報を保持し、`AdminGate`が`AdminLayout`+実際のページ(`<Outlet/>`)を表示する。既定の遷移先は`/admin`(管理者注文一覧画面)。
- 失敗時(401等)は `.error-box` にエラーメッセージを表示し、ログイン画面のまま留まる。
- ログイン成功時は`AdminAuthProvider`のReact stateとブラウザの`sessionStorage`に認証情報を保持する。同じタブで画面をリロードした場合は保存済み認証情報を復元するため、再ログインは不要。ログアウト操作またはタブを閉じると認証情報は破棄される。保存値が不正・破損している場合も破棄してログイン画面を表示する。

### 9.1 サイドバー(AdminLayout)

ログイン後の管理画面(7〜11)は共通で`components/layout/AdminLayout.tsx`によるサイドバー付きレイアウトを持つ。shadcn/uiの`Sidebar`コンポーネント(`collapsible="icon"`)を使用。

| 要素 | 内容 |
|---|---|
| サイドバーヘッダー | 開閉トリガーボタン(`SidebarTrigger`) |
| メニューグループ「メニュー」 | 注文管理(`/admin`) |
| メニューグループ「マスタ管理」 | 商品カテゴリ(`/admin/master/product-categories`)・商品マスタ(`/admin/master/products`) |
| サイドバーフッター | 更新情報(`/admin/updates`)とログアウトボタン。更新情報は業務メニューではなく補助情報としてログアウト直上に配置 |

- サイドバーは開閉可能(`collapsible="icon"`): トリガーボタンでアイコンのみの折りたたみ表示と、ラベル付きの通常表示を切り替えられる。折りたたみ時は各メニュー項目にツールチップでラベルを表示。
- 現在のURLに応じて、該当するメニュー項目がハイライト(`isActive`)される。
- サーバー側のセッション破棄処理は無い(そもそもサーバー側にセッション概念が無い)。

## 10. 管理者注文一覧画面(features/admin-orders)

- URL: `/admin`
- 目的: 全注文の概要確認・検索を行い、選択した注文の詳細画面へ移動する。

### 10.1 検索条件(検索バー)

| 項目 | 入力種別 | 説明 |
|---|---|---|
| キーワード | テキスト | 顧客名・メールアドレス・電話番号・商品名・注文IDのいずれかに部分一致(大文字小文字区別なし、`ILIKE`) |
| ステータス | セレクトボックス | 「すべて」または5ステータスのいずれか |
| 期間(開始) | 日付(date input) | `createdAt >= 指定日 00:00:00.000Z` |
| 期間(終了) | 日付(date input) | `createdAt <= 指定日 23:59:59.999Z` |
| 完了分も表示 | チェックボックス | 初期値OFF。ONの場合、検索結果に`completed`の注文を含める |

- 「検索」ボタンで `GET /orders` にクエリパラメータを付けて再取得。
- 「条件をリセット」で検索条件を全てクリアして再取得。
- 初期表示およびチェックOFFの検索では完了注文を除外する。ONの場合は`includeCompleted=true`を送る。ステータスで「完了」を選んだ場合はチェックも自動的にONになり、チェックをOFFへ戻すと「完了」のステータス指定を解除する。
- 検索条件はAND結合。日付はJST変換等は行わず、UTC基準で日付境界を作っている点に注意(サーバーのタイムゾーン設定に依存)。

### 10.2 注文一覧表示

一覧はカード(行)形式で、1注文につき以下を表示する。

| 表示項目 | 内容 |
|---|---|
| ステータス | 色付きバッジで現在値を表示(一覧では変更不可) |
| 商品 | 「商品」ラベル付きで商品名を表示 |
| 注文者 | 「注文者」ラベル付きで顧客名を表示 |
| 数量 | 「数量」ラベル付きで表示 |
| 受付日時 | 「受付日時」ラベル付き、`toLocaleString('ja-JP')`で表示 |
| 注文ID | 「注文ID」ラベル付きでUUID全体を表示 |
| 詳細導線 | 「詳細を見る →」を表示 |

カード全体がbutton要素で、クリックまたはキーボード操作により`/admin/orders/:orderId`へ遷移する。備考、連絡先、添付、支払いURL操作は一覧から外し、注文詳細画面に集約する。一覧の先頭に件数を表示し、0件時は空状態メッセージを表示する。

### 10.3 CSV出力

- 検索領域の下に「CSV出力」ボタンを表示し、現在画面に取得済みの検索結果だけをフロント側でCSVへ整形してダウンロードする。追加のAPIリクエストは行わない。
- 読み込み中、または検索結果が0件の場合はボタンを非活性にする。
- ファイル名はJSTの出力日時を使った`orders_YYYYMMDD_HHmmss.csv`とする。
- Excelで日本語が文字化けしにくいようUTF-8 BOMを付け、改行コードはCRLF、全セルをダブルクォートで囲む。値中のダブルクォート・カンマ・改行をCSV仕様に従って処理する。
- 顧客入力値が`=`・`+`・`-`・`@`で始まる場合は先頭にアポストロフィを付け、表計算ソフトで数式として実行されることを防ぐ。

| CSVカラム | 出力内容 |
|---|---|
| 注文ID | `id` |
| 注文者 | `customerName` |
| メールアドレス | `customerEmail` |
| 電話番号 | `customerPhone`。未入力時は空欄 |
| 商品 | `productName` |
| 個数 | `quantity` |
| 値段 | 注文時点の総額`totalPrice` |
| ステータス | 現在のステータスを日本語ラベルで出力 |
| 注文日時 | `createdAt`をJSTの年月日時分秒に変換 |
| ご要望 | `notes`。未入力時は空欄 |
| 添付ファイル | `fileNames`が1件以上なら「あり」、0件なら「なし」 |

### 10.4 ステータスバッジの色分け

| ステータス値 | 表示ラベル | バッジ色クラス |
|---|---|---|
| `new` | 新規注文 | `badge-new`(オレンジ系) |
| `reviewing` | 内容確認中 | `badge-reviewing`(黄色系) |
| `payment_link_sent` | メール送信済み | `badge-sent`(緑系) |
| `completed` | 完了 | `badge-completed`(青系) |
| `cancelled` | キャンセル | `badge-cancelled`(グレー系) |

一覧では表示専用。ステータス変更は注文詳細画面で行う。

## 11. 注文詳細画面(features/order-detail)

- URL: `/admin/orders/:orderId`
- レイアウト幅: 注文一覧画面と同じ`.page-wide`。固定幅ではなく、利用可能領域から共通の左右余白を引いた可変幅。
- 画面表示時に`GET /orders/:id`を呼び、注文情報・顧客情報・備考・添付ファイル名・現在の支払いURLを取得する。
- 注文情報は注文ID、受付日時、商品、数量、注文者、メールアドレス、電話番号をラベル付きで表示する。
- ステータスセレクトを変更すると`PATCH /orders/:id/status`を呼ぶ。変更中はセレクトを非活性にし、成功・失敗メッセージを表示する。
- 備考は改行を維持して表示し、未入力時は「備考はありません。」と表示する。

### 11.1 添付ファイル確認

- `fileNames`の各要素について、認証付きで`GET /orders/:id/files/:fileIndex`を取得し、ブラウザ内の一時Blob URLへ変換する。
- JPEG/PNG/GIF/WebPは画像としてカード内にプレビュー、PDFは埋め込み表示、それ以外はFILE表示とする。HEIC/HEIFはAPI側でJPEGへ変換したBlobを画像としてプレビューする。各カードからファイルを開ける。
- 添付が無い場合は「添付ファイルはありません。」、個別取得に失敗した場合はそのカード内にエラーを表示する。Blob URLは画面離脱時に破棄する。

### 11.2 支払いリンク送信

- URL入力欄は既存の`paymentLink`があれば初期表示する。空欄時は送信ボタンを非活性にする。
- 「支払いURLを送信」で`POST /orders/:id/send-payment-link`を呼び、成功時にステータスを`payment_link_sent`へ更新する。送信済み注文では「再送信」表記となり、URL変更・再送信も可能。

## 12. 商品カテゴリ管理画面(features/product-categories)

- URL: `/admin/master/product-categories`
- 目的: 商品カテゴリ(`product_categories`テーブル)の一覧確認・登録・編集・削除を行う。
- レイアウト幅: `.page-master.master-management-page`。max-width上限は設けず、管理領域の左右に`clamp(16px, 2.5vw, 32px)`の余白を確保した利用可能幅へ収める。商品マスタ管理画面と同一の共通コンポーネントとCSSを使用する。

### 12.1 検索

- 検索バー(`.filter-bar`)に「カテゴリ名で検索」のキーワード入力欄を1つ表示。
- サーバーには問い合わせず、`GET /product-categories`で取得済みの一覧をフロント側でその場で絞り込む(部分一致、大文字小文字区別なし)。カテゴリ数が少数である前提のため、注文一覧のような送信ボタン式の検索ではなく入力するたびに即座に反映される。
- 絞り込み結果が0件の場合は「該当するカテゴリがありません。」を表示(元々0件の場合と検索でヒットしない場合を区別しない)。
- 画面幅560px以下では商品マスタ・注文管理の検索欄と同じく、検索項目を1列の縦並びにして入力欄を利用可能幅いっぱいに表示する。

### 12.2 一覧表示

- shadcn/uiの`Table`でID・カテゴリ名・操作(編集/削除)列を表示。
- 共通のテーブルviewport内で最低幅560pxを維持し、ID 120px・カテゴリ名280px・操作160pxの最小幅を確保する。viewportが560px未満の場合はviewport内だけを横スクロールさせる。
- 読み込み中は「読み込み中...」を表示。
- 画面表示時に`GET /product-categories`を呼び出して一覧取得。

### 12.3 新規作成・編集

- ヘッダー右の「新規作成」ボタン、または各行の「編集」リンクでダイアログ(shadcn `Dialog`)を開く。
- 入力項目はカテゴリ名(必須)とカテゴリ画像(任意、画像ファイル・5MBまで)。HEIC/HEIFはサーバー側でJPEGへ変換する。編集時は現在画像のプレビューを表示する。
- ダイアログの最大幅、画面端の余白、内側padding、縦スクロール、セーフエリア、スマホ幅のボタン配置は商品マスタと同一の`MasterFormDialogLayout`で制御する。
- 保存ボタン押下で、新規作成時は`POST /product-categories`、編集時は`PATCH /product-categories/:id`を呼び出し、画像を選択した場合は続けて`PUT /product-categories/:id/image`を呼ぶ。未選択時は現在画像を維持する。
- マスタ保存と画像保存は別リクエストでトランザクションではない。新規作成後の画像アップロードだけが失敗した場合、カテゴリ自体は作成済みで画像未登録の状態になる。
- 成功するとダイアログを閉じて一覧を再取得。失敗時はダイアログを閉じずに`.error-box`でエラーメッセージを表示する(例: 名前が空)。

### 12.4 削除

- 各行の「削除」リンクでshadcn `AlertDialog`による確認ダイアログを表示。
- 「削除する」を押すと`DELETE /product-categories/:id`を呼び出す。
- **このカテゴリに属する商品が1件でも残っている場合は削除できない**(DBの外部キー制約 `ON DELETE RESTRICT` による)。失敗時はダイアログを閉じずに`.error-box`でエラーメッセージ(「このカテゴリに属する商品が存在するため削除できません。先に商品のカテゴリを変更するか、商品を削除してください」)を表示する。

## 13. 商品マスタ管理画面(features/product-master)

- URL: `/admin/master/products`
- 目的: 商品(`products`テーブル)の一覧確認・登録・編集・削除を行う。
- レイアウト幅: `.page-master.master-management-page`。商品カテゴリ管理画面と同一の共通コンポーネントとCSSを使い、左右余白と利用可能幅を揃える。

### 13.1 検索

検索バー(`.filter-bar`)に2つの独立した条件を表示(AND結合)。

| 項目 | 入力種別 | 説明 |
|---|---|---|
| 商品名で検索 | テキスト | 商品名に部分一致(大文字小文字区別なし) |
| カテゴリで検索 | セレクトボックス(shadcn `Select`) | 「すべて」または`GET /product-categories`で取得したカテゴリの中から1件を選択し、完全一致で絞り込む |

- 12.1と同様、サーバーには問い合わせず取得済みの一覧をフロント側でその場で絞り込む。
- 絞り込み結果が0件の場合は「該当する商品がありません。」を表示。
- 画面幅560px以下では注文管理の検索欄と同じく、検索項目を1列の縦並びにして各入力欄を利用可能幅いっぱいに表示する。

### 13.2 一覧表示

- shadcn/uiの`Table`でID・商品名・カテゴリ名・参考価格・操作(編集/削除)列を表示。
- 商品マスタ画面は管理領域の左右に`clamp(16px, 2.5vw, 32px)`の余白を確保し、その内側をテーブルviewportの最大幅とする。テーブルは最低幅840pxを維持し、ID 180px・商品名220px・カテゴリ160px・参考価格130px・操作150pxの最小幅を確保する。viewportが840px未満の場合はviewport内だけを横スクロールさせ、ページ全体の横スクロールやセルの潰れ・重なりを防ぐ。
- 画面表示時に`GET /products`(商品一覧、カテゴリ名を含む)と`GET /product-categories`(編集フォームのカテゴリ選択肢用)を並行取得する。
- カテゴリが1件も無い場合、「新規作成」ボタンが非活性になり、「先に商品カテゴリを1件以上作成してください。」という案内を表示する(商品には必ずカテゴリが必要なため)。

### 13.3 新規作成・編集

- ダイアログ(shadcn `Dialog`)形式。入力項目:
- ダイアログ幅は最大640pxかつ画面左右に最低12px(画面幅560px以下では8px)の余白を残し、`svh`を用いてSafariのブラウザUIを除いた画面内に高さを収める。内容が画面高を超える場合はダイアログ本体をタッチ操作で縦スクロールできる。保存・キャンセル操作は固定せず入力項目に続く最下部へ配置し、画面幅560px以下では画像プレビュー付き編集でも両方を同時に確認できるよう横1列で表示する。下端には端末のセーフエリアを加味した余白を確保する。

| 項目 | 入力種別 | 必須 | 備考 |
|---|---|---|---|
| ID | テキスト | ○(新規作成時のみ入力可) | 半角英小文字・数字・ハイフンのみ。URLに使われるスラッグで、**作成後は変更不可**(編集時は読み取り専用表示) |
| 商品名 | テキスト | ○ | |
| 説明 | テキストエリア | ○ | |
| 参考価格(円) | 数値 | ○ | 0以上 |
| カテゴリ | セレクトボックス(shadcn `Select`) | ○ | `GET /product-categories`で取得した一覧から選択 |
| 商品画像 | ファイル | 任意 | 画像ファイル、5MBまで。HEIC/HEIFはサーバー側でJPEGへ変換。編集時は現在画像をプレビューし、未選択なら維持 |

- 保存ボタン押下で、新規作成時は`POST /products`、編集時は`PATCH /products/:id`を呼び出し、画像を選択した場合は続けて`PUT /products/:id/image`を呼ぶ。
- マスタ保存後の画像アップロードだけが失敗した場合、商品項目の作成・更新は反映済みで画像は以前の状態(新規商品では未登録)に留まる。
- 成功するとダイアログを閉じて一覧を再取得。失敗時はダイアログを閉じずに`.error-box`でエラーを表示する(例: IDが重複、指定したカテゴリが存在しない)。

### 13.4 削除

- 各行の「削除」リンクでshadcn `AlertDialog`による確認ダイアログを表示。
- 「削除する」を押すと`DELETE /products/:id`を呼び出す。
- **この商品を参照している注文が1件でも残っている場合は削除できない**(DBの外部キー制約 `ON DELETE RESTRICT` による)。失敗時はダイアログを閉じずに`.error-box`でエラーメッセージを表示する。

## 14. 更新情報画面(features/update-history)

- URL: `/admin/updates`
- 目的: 管理者が現在デプロイされているアプリのバージョンと、これまでに追加・改善された機能を確認する。
- サイドバーでは業務メニューと分け、フッターのログアウトボタン直上に「更新情報」として配置する。
- 現在のバージョンは`frontend/package.json`の`version`をビルド時に読み込んで表示する。
- 更新履歴は`features/update-history/data/updateHistory.ts`の静的配列を新しい順で表示する。日付・バージョン・変更概要を共通の1行形式で表示し、行を選択するとダイアログで詳しい変更内容を確認できる。APIやDBは使用しない。
- ユーザーに見える機能追加・修正時は、コード変更と同時に該当バージョンの`changes`へ1項目追加する。
- `.page-wide`を使用して他の管理画面と横幅を揃える。現在のバージョンは見出し横のバッジで表示し、スマホ幅でも各履歴は省略表示を使って1行を維持する。

## 15. 未実装・既知の制約(画面まわり)

- 管理者ログイン状態は同一タブの`sessionStorage`に保持されるためリロードでは維持されるが、タブを閉じると破棄される。JWTやサーバーセッションによる認証ではない。
- 注文の削除・キャンセル理由入力などの機能は無い(ステータスを`cancelled`にするだけ)。
- ページネーションは無く、`GET /orders`・`GET /products`・`GET /product-categories`の結果を全件一覧表示する。
- 商品マスタ・商品カテゴリの一括操作(複数選択削除など)は無い。
- 商品カテゴリの並び順を変更するUIは無い(常にid昇順)。
